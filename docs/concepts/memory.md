# 기억 시스템 (Memory)

이 문서에서는 OpenClaw의 핵심 개념을 쉽게 이해하는 방법을 배웁니다.

## 📌 이 문서에서 배우는 것
- 1. `MEMORY.md`
- 2. `memory/` 디렉터리
- 3. flush

걱정하지 마세요, 하나씩 따라하면 됩니다! 😊


OpenClaw의 메모리는 “대화가 길어도 안 까먹게 하는 기능” 이상입니다. 최신 기준에서 메모리는 **`MEMORY.md` + `memory/` + flush + search**가 결합된 계층으로 보는 것이 가장 정확합니다.

## 1. `MEMORY.md`

가장 상위의 요약 기억입니다. 보통 사용자 프로필, 작업 선호, 장기 프로젝트 정보, 운영 상수처럼 **자주 다시 참조해야 하는 핵심 사실**이 들어갑니다.

특징:

- 사람이 읽고 직접 편집하기 쉬움
- 가장 압축된 “정본 요약” 역할
- 에이전트 bootstrap에 자주 반영됨

## 2. `memory/` 디렉터리

`MEMORY.md`가 핵심 요약이라면, `memory/`는 더 세분화된 기록을 담는 계층입니다.

예:

- 날짜별 로그
- 프로젝트별 기록
- 특정 사건/의사결정 메모
- 누적 학습 기록

즉, `memory/`는 **더 풍부하지만 덜 압축된 기억 저장소**입니다.

## 3. flush

긴 세션이나 큰 작업에서는 모든 세부 맥락을 항상 세션 안에 둘 수 없습니다. 그래서 일부 내용을 요약·정리해 메모리 계층으로 **flush**하는 흐름이 중요합니다.

flush의 목적:

- 세션 크기 축소
- 중요한 사실의 장기 보존
- 이후 검색 가능성 확보

즉, flush는 “세션에서 메모리로 올리는 승격 과정”이라고 볼 수 있습니다.

## 4. search

기억은 저장만 하면 끝이 아닙니다. 필요할 때 다시 찾아와야 의미가 있습니다. OpenClaw의 메모리 체계는 검색(retrieval/search)과 함께 작동합니다.

검색은 보통 다음 질문에 답합니다.

- 지금 요청과 관련된 과거 사실이 무엇인가?
- 이전에 비슷한 작업을 어떻게 했는가?
- 사용자의 선호/금지사항이 무엇인가?

이 회수 과정이 있어야 메모리는 실제 작업 성능으로 이어집니다.

## 세션과 메모리의 역할 분담

- **세션**: 지금 당장 진행 중인 대화/작업의 흐름
- **메모리**: 다음 대화에서도 계속 유효한 장기 사실

좋은 운영에서는 모든 것을 메모리에 넣지 않고, 모든 것을 세션에만 남기지도 않습니다. 둘 사이를 flush와 search로 연결하는 것이 핵심입니다.

## 사람이 직접 관리할 수 있다는 점

OpenClaw 메모리의 큰 장점은 사람이 읽고 고치기 쉬운 파일 기반 구조라는 점입니다. 즉, 메모리는 블랙박스 데이터베이스가 아니라 **점검·수정 가능한 운영 자산**입니다.
