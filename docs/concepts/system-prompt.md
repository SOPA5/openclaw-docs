# 시스템 프롬프트 (System Prompt)

이 문서에서는 OpenClaw의 핵심 개념을 쉽게 이해하는 방법을 배웁니다.

## 📌 이 문서에서 배우는 것
- 무엇이 합쳐지나
- 1. 기본 정체성 및 역할
- 2. bootstrap 정보

걱정하지 마세요, 하나씩 따라하면 됩니다! 😊


OpenClaw의 시스템 프롬프트는 한 장의 고정 문장이 아닙니다. 실제 런타임에서는 **skill / bootstrap / tooling / runtime 정책이 결합된 동적 구성물**에 가깝습니다.

## 무엇이 합쳐지나

최종 시스템 프롬프트는 보통 다음 계층이 합쳐져 만들어집니다.

### 1. 기본 정체성 및 역할

에이전트의 말투, 기본 임무, 우선순위 같은 상위 지침입니다.

### 2. bootstrap 정보

작업 디렉터리, 현재 날짜, 운영 환경, 핵심 규칙, 프로젝트 문맥처럼 런타임 시작 시 주입되는 정보입니다.

### 3. tool definitions

에이전트가 실제로 쓸 수 있는 도구 목록과 파라미터 규칙입니다. 이 계층이 있어야 에이전트는 “무엇을 호출할 수 있는지”를 압니다.

### 4. skill instructions

특정 작업에 맞는 전문 지침입니다. 예를 들어 GitHub 작업, 날씨 조회, 문서 게시, 코딩 에이전트 라우팅 같은 시나리오에서 행동 방식을 더 구체화합니다.

### 5. runtime policy

안전, 승인 경계, 샌드박스, 세션 정책, 보고 형식 같은 운영 규칙입니다.

## 왜 동적으로 조합하나

같은 모델이라도 상황이 다르면 필요한 지시가 완전히 달라집니다.

예:

- 일반 채팅 세션
- ACP 코딩 런타임 연결
- 브라우저 자동화 작업
- 서브에이전트 세션
- 특정 skill이 강하게 적용된 작업

모든 상황에 하나의 고정 프롬프트를 쓰면 과하거나 부족해집니다. 그래서 OpenClaw는 **현재 상황에 맞는 지시 세트**를 조립하는 방식을 택합니다.

## 시스템 프롬프트를 볼 때 흔한 오해

### 오해 1. “성격 문장”만 시스템 프롬프트다

아닙니다. OpenClaw에서는 도구 규칙과 런타임 정책이 훨씬 큰 비중을 차지합니다.

### 오해 2. skill은 프롬프트와 별개다

실제로는 skill이 시스템 지시 확장으로 작동하는 경우가 많습니다. 즉, skill은 별도 UI 개념이면서 동시에 프롬프트 구성 요소이기도 합니다.

### 오해 3. 시스템 프롬프트는 한 번 정해지면 끝이다

세션 종류, 사용 가능한 도구, bootstrap, 채널, 서브에이전트 여부에 따라 달라질 수 있습니다.

## 실무적으로 중요한 점

시스템 프롬프트는 답변 품질만 바꾸는 것이 아닙니다. 다음까지 바꿉니다.

- 어떤 도구를 우선 고려하는가
- 언제 사용자의 승인을 구하는가
- 어떤 형식으로 보고하는가
- 어떤 위험 경계를 넘지 않는가

즉, 시스템 프롬프트는 “말투 설정”이 아니라 **행동 운영체제**에 가깝습니다.
