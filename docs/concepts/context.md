# 컨텍스트 관리 (Context)

이 문서에서는 OpenClaw의 핵심 개념을 쉽게 이해하는 방법을 배웁니다.

## 📌 이 문서에서 배우는 것
- context engine이 모으는 재료
- 왜 compaction이 필요한가
- session pruning

걱정하지 마세요, 하나씩 따라하면 됩니다! 😊


OpenClaw의 컨텍스트는 단순히 “최근 대화 몇 줄”이 아닙니다. 최신 구조에서는 **context engine**이 세션 기록, 도구 출력, 시스템 정책, 메모리, 워크스페이스 단서를 함께 조합해 모델에 넘깁니다.

## context engine이 모으는 재료

대표적으로 다음이 들어갑니다.

- 최근 대화 기록
- 직전 도구 호출과 결과
- 시스템 프롬프트와 런타임 정책
- 현재 세션 메타데이터
- 관련 메모리 검색 결과
- 필요 시 워크스페이스 파일 정보

즉, 컨텍스트는 정적인 텍스트가 아니라 **현재 작업에 맞춰 조립되는 실행용 입력 묶음**입니다.

## 왜 compaction이 필요한가

에이전트가 오래 일할수록 쌓이는 정보는 빠르게 커집니다. 모든 것을 그대로 넣으면 비용도 커지고, 중요한 정보가 오히려 묻힙니다. 그래서 OpenClaw는 **compaction**을 사용합니다.

compaction의 목적:

- 중복 제거
- 오래된 세부 출력 축약
- 중요한 상태만 보존
- 모델 창(context window) 압박 완화

즉, compaction은 단순 요약이 아니라 **작업 지속성을 위한 문맥 압축 장치**입니다.

## session pruning

compaction이 “내용을 줄이는 기술”이라면, **session pruning**은 “얼마나 오래 붙들고 있을지 정하는 정책”에 가깝습니다.

pruning이 다루는 것:

- 오래된 assistant/user turn 정리
- 너무 큰 도구 출력 제거
- 오래된 세션의 하드 클리어 또는 보관
- 최근 중요 응답 몇 개 유지

이 정책 덕분에 장기 운영에서도 세션이 무한정 비대해지지 않습니다.

## context flush

긴 세션에서는 일부 맥락을 별도 요약/기록으로 **flush**해 두고, 필요할 때 다시 회수하는 패턴이 중요합니다. 이는 모든 원문을 항상 유지하는 것보다 훨씬 실용적입니다.

이 흐름은 [메모리](/concepts/memory) 시스템과도 자연스럽게 연결됩니다.

## 좋은 컨텍스트의 기준

좋은 컨텍스트는 “많은 컨텍스트”가 아닙니다. 아래 네 가지를 만족해야 합니다.

- 현재 작업과 관련 있음
- 너무 장황하지 않음
- 정책/제약을 빠뜨리지 않음
- 필요한 과거 사실은 다시 찾아올 수 있음

OpenClaw의 context engine은 바로 이 균형을 맞추기 위해 존재합니다.
