# 에이전트 런타임 (Agent Runtime)

이 문서에서는 OpenClaw의 핵심 개념을 쉽게 이해하는 방법을 배웁니다.

## 📌 이 문서에서 배우는 것
- 에이전트가 실제로 하는 일
- 도구 스트리밍
- 세션 도구와 에이전트 제어 도구

걱정하지 마세요, 하나씩 따라하면 됩니다! 😊


OpenClaw의 에이전트는 단순 텍스트 생성기가 아니라, **Gateway가 제공하는 도구와 세션을 사용해 실제 작업을 수행하는 런타임**입니다.


:::tip 💡 쉽게 이해하기
**Gateway**는 쉽게 말해 "교환대"예요. 여러 앱과 도구, AI 모델 사이를 이어주는 중앙 통로라고 생각하면 이해하기 쉽습니다.
:::

## 에이전트가 실제로 하는 일

에이전트는 보통 다음 루프를 반복합니다.

1. 사용자 요청 해석
2. 현재 세션과 컨텍스트 확인
3. 필요한 도구 호출
4. 결과를 다시 해석
5. 다음 행동 또는 최종 응답 결정

이 루프는 일반 챗봇보다 훨씬 “실행 중심”입니다.

## 도구 스트리밍

최신 OpenClaw 흐름에서 중요한 특징 중 하나는 **도구 스트리밍(tool streaming)** 입니다. 즉, 에이전트는 내부 생각이 끝난 뒤 한 번에 결과만 내놓는 것이 아니라, 도구 호출과 그 결과를 단계적으로 다룰 수 있습니다.

이 구조의 장점:

- 긴 작업 중간 상태를 더 잘 다룸
- 브라우저/exec 같은 작업형 도구와 잘 맞음
- Control UI나 API에서 응답 흐름을 더 구조적으로 표현 가능

## 세션 도구와 에이전트 제어 도구

OpenClaw에는 단순 작업 도구 외에, **다른 세션과 에이전트를 다루는 도구 계층**도 있습니다.

대표 예:

- `sessions_spawn`
- `sessions_list`
- `sessions_history`
- `sessions_yield`

이 계층 덕분에 에이전트는 단순히 “일을 직접 수행”할 뿐 아니라, **다른 작업자를 띄우고, 상태를 넘기고, 결과를 회수하는 오케스트레이터** 역할도 할 수 있습니다.

## ACP 런타임

OpenClaw는 자체 서브에이전트만이 아니라, ACP 런타임을 통해 Codex, Claude Code, Gemini CLI 같은 외부 코딩 에이전트와도 연결될 수 있습니다.

이때 중요한 점:

- ACP는 일반 서브에이전트와 같은 것이 아닙니다.
- 스폰 방식과 수명주기 관리가 다를 수 있습니다.
- Gateway는 이를 채팅 세션/스레드와 결합해 관리할 수 있습니다.

즉, OpenClaw의 에이전트 시스템은 “내장 에이전트만 있는 구조”가 아니라, **멀티런타임 라우팅 구조**입니다.

## 멀티에이전트 라우팅

하나의 요청이 항상 하나의 에이전트만 쓰는 것은 아닙니다. 상황에 따라 다음처럼 분기할 수 있습니다.

- 메인 에이전트가 직접 답변
- 리서치용 서브에이전트 위임
- 코딩 작업을 ACP 세션으로 위임
- 그룹/채널 정책에 따라 별도 격리 세션 사용

즉, 라우팅은 단순 모델 선택이 아니라 **누가 어떤 컨텍스트와 도구로 일할지 정하는 것**입니다.

## 에이전트를 설계할 때 생각할 점

- 어떤 모델을 쓸 것인가
- 어떤 도구를 허용할 것인가
- 어떤 스킬을 부트스트랩할 것인가
- 세션을 메인 공유로 할지, 격리할지
- 메모리/검색을 얼마나 강하게 연결할 것인가

이 조합이 곧 “에이전트의 성격”을 만듭니다.
