# 메모리 & 컴팩션 (Memory & Compaction)

OpenClaw는 긴 대화를 계속 이어 가야 합니다. 그런데 모델이 한 번에 볼 수 있는 양에는 한계가 있습니다.

그래서 OpenClaw는 크게 3가지를 함께 씁니다.
- 지금 대화 흐름을 잡는 **세션 메모리**
- 오래 기억할 사실을 두는 **파일 메모리**
- 너무 길어진 대화를 줄여 주는 **컴팩션**

## 📌 이 문서에서 배우는 것
- 세션 메모리가 무엇인지
- `MEMORY.md`와 `memory/*.md`의 차이
- 컴팩션이 왜 필요한지
- 컴팩션 전에 중요한 내용을 저장하는 memory flush 개념
- 무엇을 장기 기억에 넣어야 하는지

:::tip 쉽게 이해하기
- 세션 메모리 = 오늘 수업 시간에 쓰는 칠판
- `MEMORY.md` = 오래 두고 보는 반 전체 공지 노트
- `memory/*.md` = 날짜별로 적는 학급 일지
- 컴팩션 = 칠판이 꽉 차서 중요한 내용만 다시 정리해 적는 것
:::

## 세션 메모리란?

세션 메모리는 **지금 이 대화에서 이어지고 있는 흐름**입니다.

예를 들면:
- 지금 무슨 작업을 하는지
- 방금 어떤 결정을 했는지
- 다음에 무엇을 해야 하는지

이 정보는 대화가 이어질수록 점점 쌓입니다.

하지만 너무 많이 쌓이면 모델이 한 번에 보기 어려워집니다. 그래서 나중에 설명할 **컴팩션**이 필요해집니다.

## `MEMORY.md`: 장기 기억

`MEMORY.md`는 오래 남겨 둘 중요한 정보를 적는 파일입니다.

예를 들면:
- 사용자의 선호 방식
- 자주 쓰는 프로젝트 규칙
- 계속 유지되어야 하는 운영 원칙
- 다음 세션에서도 꼭 알아야 하는 사실

이 파일은 쉽게 말해 **"앞으로도 계속 기억해야 하는 핵심 노트"** 입니다.

### 여기에 넣기 좋은 것
- 매번 다시 설명하면 아까운 정보
- 자주 바뀌지 않는 규칙
- 사용자 취향이나 작업 습관
- 중요한 운영 상수

### 여기에 넣지 않는 게 좋은 것
- 오늘만 잠깐 필요한 메모
- 이미 끝난 작은 잡일
- 금방 바뀔 임시 상태
- 너무 세세한 대화 로그 전체

:::tip
`MEMORY.md`는 많이 적는 곳이 아니라 **정말 중요한 것만 남기는 곳**입니다. 너무 많이 넣으면 오히려 찾기 어려워집니다.
:::

## `memory/*.md`: 일별 기억

`memory/*.md`는 날짜별로 남기는 기록입니다.

예를 들면:
- 오늘 무엇을 했는지
- 어떤 문제가 있었는지
- 어떤 결정을 했는지
- 내일 이어서 볼 포인트가 무엇인지

쉽게 말하면 **작업 일기장**입니다.

`MEMORY.md`가 큰 원칙과 장기 정보라면,
`memory/*.md`는 **오늘 있었던 중요한 일의 요약**에 더 가깝습니다.

## 컴팩션이란?

컴팩션은 대화가 너무 길어졌을 때, 앞부분을 **짧고 중요한 요약으로 다시 정리하는 과정**입니다.

왜 필요할까요?
- 모델이 볼 수 있는 토큰 수에는 한계가 있기 때문
- 긴 대화를 그대로 다 들고 가면 느려지고 비싸질 수 있기 때문
- 중요한 것만 남기고 계속 작업해야 하기 때문

즉, 컴팩션은 "다 지우기"가 아니라 **핵심만 남기고 가볍게 다시 접기**입니다.

## 컴팩션은 언제 일어날까?

보통 대화가 길어져서 컨텍스트가 빡빡해질 때 일어납니다.

그때 OpenClaw는 앞부분 대화를 통째로 계속 들고 가는 대신,
중요한 사실과 진행 상태를 요약해 다음 흐름에 넘겨줍니다.

## 컴팩션 설정 예시

OpenClaw에서는 컴팩션에 이런 설정이 들어갈 수 있습니다.

- `mode: safeguard`
- `model`: 컴팩션에 사용할 모델
- `reserveTokensFloor`: 남겨 둘 최소 토큰 바닥선

### 뜻을 아주 쉽게 풀면

#### `mode: safeguard`
너무 공격적으로 잘라내지 않고, **안전하게 요약하자**는 뜻에 가깝습니다.

#### `model`
어떤 모델이 요약 작업을 맡을지 정합니다.

#### `reserveTokensFloor`
앞으로 이어질 대화를 위해 **최소한 이 정도 공간은 남겨 두자**는 기준입니다.

:::tip
컴팩션은 성능을 아끼기 위한 장치이기도 하지만, 더 중요한 목적은 **긴 작업이 중간에 흐트러지지 않게 하는 것**입니다.
:::

## memory flush란?

memory flush는 컴팩션이 일어나기 전에, 중요한 내용을 메모리 파일로 **먼저 빼서 저장하는 동작**을 뜻합니다.

쉽게 말하면:
1. 대화가 길어짐
2. 곧 요약이 필요함
3. 중요한 사실을 `MEMORY.md`나 `memory/*.md`로 저장함
4. 그다음 안심하고 컴팩션함

이 과정을 하면 중요한 사실이 대화 본문에서 사라져도, **파일 기억에는 남아 있게** 됩니다.

## 실전 팁: 무엇을 `MEMORY.md`에 넣어야 할까?

넣으면 좋은 것:
- 사용자가 좋아하는 답변 스타일
- 자주 반복되는 금지 규칙
- 프로젝트의 핵심 구조
- 다음번에도 꼭 유지해야 하는 운영 방식

안 넣는 게 좋은 것:
- 오늘만 필요한 임시 TODO
- 아주 짧은 실험 결과
- 이미 끝난 세부 로그
- 검색 결과 원문 전체

### 고르는 쉬운 기준

이 질문을 해 보면 됩니다.

> "이 정보를 다음 주에도 다시 꼭 써야 할까?"

- **예** → `MEMORY.md` 후보
- **아마 오늘만 필요** → 세션 또는 `memory/*.md`

## 한 번에 정리하면

- 세션 메모리 = 지금 대화 흐름
- `MEMORY.md` = 오래 남길 핵심 기억
- `memory/*.md` = 날짜별 작업 기록
- 컴팩션 = 너무 긴 대화를 짧게 정리하는 일
- memory flush = 컴팩션 전에 중요한 내용부터 저장하는 안전장치

## ✅ 성공 확인
- 세션 메모리가 무엇인지 설명할 수 있습니다.
- `MEMORY.md`와 `memory/*.md`의 차이를 이해했습니다.
- 컴팩션이 왜 필요한지 말할 수 있습니다.
- `mode`, `model`, `reserveTokensFloor`가 어떤 느낌의 설정인지 알게 되었습니다.
- 무엇을 장기 기억에 넣어야 하는지 판단 기준을 얻었습니다.
