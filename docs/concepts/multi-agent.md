# 멀티 에이전트 (Multi-agent)

멀티 에이전트는 **한 명이 다 하지 않고, 여러 작업자가 역할을 나눠서 함께 일하는 방식**입니다.

예를 들어 큰 숙제가 있으면:
- 한 명은 자료 조사
- 한 명은 만들기
- 한 명은 오류 찾기
를 맡을 수 있죠.

OpenClaw도 비슷합니다.

## 📌 이 문서에서 배우는 것
- 왜 여러 에이전트가 필요한지
- 에이전트 목록을 보고 추가하는 기본 명령
- 메인과 워커의 역할 차이
- `sessions_spawn`으로 서브에이전트를 띄우는 이유
- ACP 런타임과 연결되는 그림

:::tip 쉽게 이해하기
멀티 에이전트는 **축구팀**과 비슷합니다. 골키퍼, 수비수, 공격수가 다 역할이 다르듯, 에이전트도 같은 능력을 똑같이 쓰기보다 역할을 나눌 때 더 강해집니다.
:::

## 왜 여러 에이전트가 필요할까?

큰 작업은 보통 한 가지 일만 있지 않습니다.

예를 들면 문서 하나를 만들 때도:
- 사실 조사
- 초안 작성
- 문장 다듬기
- 오류 점검
이 함께 들어갑니다.

이걸 한 에이전트가 다 하면 가능은 하지만,
- 느려질 수 있고
- 앞에서 한 걸 뒤에서 잊기 쉽고
- 검토가 약해질 수 있습니다.

그래서 OpenClaw는 필요하면 **작업을 나눠 맡기는 구조**를 씁니다.

## 에이전트 목록 보기

지금 어떤 에이전트가 준비되어 있는지 보려면:

```bash
openclaw agents list
```

이 명령은 보통 다음을 확인할 때 씁니다.
- 어떤 에이전트가 등록되어 있는가?
- 이름이 무엇인가?
- 어떤 역할로 쓰는가?

## 에이전트 추가하기

새 에이전트를 등록할 때는 보통 이렇게 시작합니다.

```bash
openclaw agents add
```

상황에 따라 이름, 모델, 역할 같은 정보를 넣어 새 작업자를 추가할 수 있습니다.

:::tip
처음부터 에이전트를 너무 많이 만들 필요는 없습니다. 보통은 **main + worker 몇 개**만 있어도 충분합니다.
:::

## 메인(main) vs 워커(worker)

멀티 에이전트에서 자주 나오는 가장 중요한 구분입니다.

### main
- 사용자와 직접 이야기하는 창구
- 전체 계획을 잡음
- 결과를 합쳐서 최종 답을 만듦

### worker
- 메인이 맡긴 작은 일을 수행
- 조사, 구현, 테스트, 검토처럼 특정 역할을 맡음
- 보통 최종 사용자 응답은 메인이 담당

즉,
- **main은 감독**
- **worker는 선수**
같은 느낌입니다.

## 워크스페이스 격리

여러 에이전트가 같이 일하면 파일이 섞이거나 충돌할 수 있습니다.

그래서 작업할 때는 **워크스페이스를 어떻게 나눌지**가 중요합니다.

예를 들면:
- 같은 프로젝트를 보되 역할만 나누기
- 임시 폴더에서 안전하게 실험하기
- 리뷰용 에이전트는 읽기 중심으로만 쓰기

워크스페이스 격리는 쉽게 말해 **각자 책상 구역을 나눠 주는 것**입니다.

이렇게 하면 누가 무엇을 바꿨는지 추적하기 쉽고, 실수도 줄어듭니다.

## 에이전트 라우팅

라우팅은 "이 일은 누구에게 보낼까?"를 정하는 것입니다.

예를 들면:
- 사용자 질문 이해와 최종 답변 → main
- 자료 조사 → research worker
- 코드 작성 → build worker
- 품질 점검 → QA worker

좋은 라우팅은 **적절한 사람에게 적절한 일을 보내는 것**입니다.

## 서브에이전트 스폰: `sessions_spawn`

OpenClaw에서는 새 작업자를 띄울 때 `sessions_spawn`을 씁니다.

이 도구는 쉽게 말해 **"이 일만 따로 맡아줘"라고 새 세션을 열어 주는 버튼**입니다.

왜 중요할까요?
- 큰 일을 잘게 나눌 수 있음
- 병렬로 진행할 수 있음
- 메인이 전체 그림을 놓치지 않게 도와줌

예를 들면 메인이 이렇게 생각할 수 있습니다.
- "너는 리서치 맡아"
- "너는 초안 작성 맡아"
- "너는 품질 검토 맡아"

## ACP 런타임 연동

OpenClaw는 자체 서브에이전트뿐 아니라 ACP 런타임과도 연결될 수 있습니다.

이 말은, 어떤 작업은 다른 코딩 에이전트 런타임에 맡겨서 처리할 수도 있다는 뜻입니다.

예를 들면:
- Codex 계열 에이전트에게 코드 작업 맡기기
- Claude Code 계열 흐름과 연결하기
- Gemini CLI 계열 작업 라우팅하기

즉, 멀티 에이전트는 OpenClaw 안의 팀만이 아니라 **바깥 전문 작업자와 협업하는 다리**가 될 수도 있습니다.

## 실전 예시: 메인 + 리서치 + 빌드 + QA

문서나 앱 기능 하나를 만든다고 생각해 봅시다.

### 1) 메인
- 사용자 목표 이해
- 전체 순서 정하기
- 마지막 결과 정리

### 2) 리서치 에이전트
- 공식 문서 확인
- 필요한 사실 정리
- 근거 수집

### 3) 빌드 에이전트
- 실제 파일 수정
- 기능 추가
- 초안 작성

### 4) QA 에이전트
- 빠진 내용 찾기
- 오류 점검
- 표현이 이해 쉬운지 확인

이 구조의 장점은 **역할이 겹치지 않고, 각자 잘하는 일에 집중할 수 있다는 것**입니다.

## 언제 멀티 에이전트를 쓰면 좋을까?

잘 맞는 경우:
- 작업이 길다
- 조사와 구현이 따로 놀 수 있다
- 검토가 중요하다
- 여러 갈래를 병렬로 처리하고 싶다

굳이 안 써도 되는 경우:
- 한 줄 수정 같은 아주 작은 일
- 나누면 오히려 설명 비용이 더 큰 일
- 파일 충돌 위험이 큰 매우 좁은 작업

## ✅ 성공 확인
- 왜 여러 에이전트가 필요한지 설명할 수 있습니다.
- `openclaw agents list`와 `openclaw agents add`의 역할을 구분할 수 있습니다.
- main과 worker의 차이를 이해했습니다.
- `sessions_spawn`이 서브에이전트를 띄우는 핵심 도구라는 걸 알게 되었습니다.
- 메인 + 리서치 + 빌드 + QA 구조를 머릿속으로 그릴 수 있습니다.
