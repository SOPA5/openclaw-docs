# 세션 관리 (Sessions)

이 문서에서는 OpenClaw의 핵심 개념을 쉽게 이해하는 방법을 배웁니다.

## 📌 이 문서에서 배우는 것
- main / group isolation
- 메인 세션
- 그룹 격리 세션

걱정하지 마세요, 하나씩 따라하면 됩니다! 😊


세션은 OpenClaw의 대화 단위이자 작업 단위입니다. 하지만 최신 구조에서 세션은 단순 채팅방 기록이 아니라, **격리 수준·문맥 지속성·에이전트 오케스트레이션을 함께 담는 실행 컨테이너**에 가깝습니다.


:::tip 💡 쉽게 이해하기
**Session**은 하나의 대화 흐름이에요. 같은 대화 안에서 어떤 맥락이 이어지는지 묶어서 기억하는 단위라고 보면 됩니다.
:::

## main / group isolation

OpenClaw는 모든 대화를 완전히 똑같이 다루지 않습니다. 대표적으로 다음 구분이 중요합니다.

### 메인 세션

개인 DM이나 핵심 대화 흐름은 메인 세션으로 이어질 수 있습니다. 이렇게 하면 기억과 문맥이 잘 이어집니다.

### 그룹 격리 세션

그룹 채팅은 개인 DM과 같은 문맥을 그대로 섞지 않도록 **격리된 세션**으로 두는 것이 일반적입니다. 이 구조는 프라이버시와 문맥 오염 방지에 모두 도움이 됩니다.

즉, 세션 정책은 단순 편의 기능이 아니라 **보안과 품질 둘 다에 영향**을 줍니다.

## 세션이 담는 것

세션은 보통 아래를 함께 가집니다.

- 사용자/채널 맥락
- 최근 메시지 기록
- 도구 호출 흔적
- 요약/compaction 결과
- 필요 시 서브에이전트/ACP 연계 정보

## pruning과 유지 전략

세션이 길어지면 무조건 좋은 것이 아닙니다. 그래서 OpenClaw는 pruning 정책을 사용합니다.

대표 목적:

- 너무 오래된 턴 제거
- 대형 도구 출력 축약
- 최근 핵심 응답 유지
- 저장 공간 및 모델 비용 관리

이 정책은 [컨텍스트](/concepts/context)와 밀접하게 연결됩니다.

## 세션 도구

OpenClaw에는 세션을 프로그래밍 방식으로 다루는 도구가 있습니다.

- `sessions_spawn`: 새 작업 세션/서브에이전트/ACP 세션 생성
- `sessions_list`: 세션 목록 조회
- `sessions_history`: 특정 세션 기록 확인
- `sessions_yield`: 현재 턴 종료 후 결과 대기

이 계층 덕분에 에이전트는 세션을 단순 소비하지 않고 **직접 생성·관리·위임**할 수 있습니다.

## 좋은 세션 설계 원칙

- 개인 문맥과 그룹 문맥은 분리
- 너무 긴 세션은 compaction/pruning으로 관리
- 장기 사실은 메모리 계층으로 올리기
- 무거운 작업은 새 세션 또는 서브에이전트로 분리
