# 도구 샌드박스 (Sandbox)

이 문서에서는 OpenClaw 도구를 안전하고 효과적으로 쓰는 방법을 배웁니다.

## 📌 이 문서에서 배우는 것
- 최신 관점의 핵심 용어
- tool policy
- sandbox mode

걱정하지 마세요, 하나씩 따라하면 됩니다! 😊


도구 샌드박스는 에이전트가 도구를 쓸 때의 **최신 tool policy 체계**를 설명하는 문서입니다. 핵심은 “허용/차단 목록 몇 개”가 아니라, **도구 종류·실행 환경·권한 상승·승인 경계**를 함께 설계하는 것입니다.


:::tip 💡 쉽게 이해하기
**Sandbox**는 위험한 작업을 본체와 떨어진 안전한 공간에서 실행하는 격리 상자예요. 실험실처럼 따로 떼어 두는 개념이라고 생각하면 됩니다.
:::

## 최신 관점의 핵심 용어

### tool policy

어떤 도구를 허용할지, 어떤 파라미터를 제한할지, 어떤 실행이 승인 대상인지 정하는 정책 계층입니다.

### sandbox mode

도구를 로컬에서 실행할지, 더 격리된 환경에서 실행할지 정하는 계층입니다.

### elevated

호스트에서 더 높은 권한으로 실행하는 경우를 가리킵니다. 일반 실행과 분리해서 봐야 합니다.

## tool policy가 막는 것

- 무분별한 파일 쓰기/삭제
- 고위험 셸 실행
- 불필요한 네트워크 접근
- 사용자 프로필/민감 자원 접근
- 승인 없는 elevated 실행

## 정책을 설계하는 기본 질문

- 이 에이전트가 정말 `exec`가 필요한가?
- `browser`는 격리 프로필만 써도 되는가?
- `message`는 외부 발송 권한이 필요한가?
- `write`와 `apply_patch` 중 어느 정도만 허용할 것인가?
- elevated 실행은 완전히 막을 수 있는가?

이 질문이 곧 최소 권한 설계입니다.

## 도구별로 체감 위험이 다릅니다

낮은 편:

- `read`
- `web_search`
- `web_fetch`

중간:

- `write`
- `edit`
- `browser`
- `image`

높은 편:

- `exec`
- `process`
- `message`
- 사용자 프로필을 쓰는 브라우저 자동화
- `elevated` 실행

같은 에이전트라도 작업 목적에 따라 허용 집합을 다르게 두는 것이 좋습니다.

## 운영 팁

- 기본은 좁게, 필요할 때만 넓히기
- 공유 서버일수록 sandbox mode를 엄격하게
- 외부 발송/삭제/권한 상승은 승인 경계 명확화
- 정책은 Gateway (중앙 통로) 보안과 따로 보지 않기
