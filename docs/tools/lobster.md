# Lobster: 규격화된 워크플로우 엔진

이 문서에서는 OpenClaw 도구를 안전하고 효과적으로 쓰는 방법을 배웁니다.

## 📌 이 문서에서 배우는 것
- 왜 별도 위치가 필요한가
- 어떤 작업에 적합한가
- 도구/스킬과의 관계

걱정하지 마세요, 하나씩 따라하면 됩니다! 😊


Lobster는 최신 OpenClaw 문맥에서 “기본 내장 도구 하나”라기보다, **워크플로우/자동화 플러그인 계층에 위치한 구성요소**로 이해하는 것이 더 정확합니다. 즉, Lobster는 자유형 채팅을 보조하는 것이 아니라, **반복 가능한 절차를 구조화된 흐름으로 만들기 위한 자동화 레이어**입니다.

## 왜 별도 위치가 필요한가

일반 도구는 한 번의 행동을 수행합니다.

- 파일 읽기
- 브라우저 클릭
- 메시지 보내기
- 셸 명령 실행

반면 Lobster가 다루는 것은 더 큰 단위입니다.

- 여러 단계를 가진 워크플로우
- 상태 전이가 있는 자동화
- 중간 승인(human-in-the-loop)
- 재시도와 체크포인트
- 구조화된 입력/출력

즉, Lobster는 “도구”라기보다 **자동화 오케스트레이션 플러그인**에 가깝습니다.

## 어떤 작업에 적합한가

- 규칙이 반복되는 운영 자동화
- 여러 외부 시스템을 잇는 워크플로우
- 승인 단계를 포함하는 반자동 프로세스
- 사람이 매번 같은 지시를 반복하기 싫은 작업

## 도구/스킬과의 관계

Lobster는 다른 도구를 대체하지 않습니다. 오히려:

- **도구**는 개별 실행 단위
- **스킬**은 사용 지침
- **Lobster 같은 워크플로우 플러그인**은 절차 전체를 구조화

하는 식으로 역할이 나뉩니다.

## 언제 Lobster를 고려하면 좋은가

아래 조건이 보이면 Lobster 계열 접근이 잘 맞습니다.

- 같은 작업을 자주 반복한다
- 매번 사람 승인이 일부 단계에서만 필요하다
- 단일 프롬프트보다 상태 기계(stateful flow)가 낫다
- 실패 후 재개가 중요하다
