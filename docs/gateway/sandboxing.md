# 운영 환경에서의 샌드박싱 (Sandboxing)

이 문서에서는 OpenClaw Gateway가 어떻게 동작하는지 배웁니다.

## 📌 이 문서에서 배우는 것
- 세 가지 축으로 이해하기
- 1. sandbox mode
- 2. tool policy

걱정하지 마세요, 하나씩 따라하면 됩니다! 😊


샌드박싱은 “에이전트가 도구를 쓸 수 있느냐”만이 아니라, **어디서**, **어떤 정책으로**, **얼마나 높은 권한으로** 실행되느냐를 정하는 층입니다. Gateway 관점에서 보면 샌드박싱은 도구 안전장치의 핵심입니다.


:::tip 💡 쉽게 이해하기
**Sandbox**는 위험한 작업을 본체와 떨어진 안전한 공간에서 실행하는 격리 상자예요. 실험실처럼 따로 떼어 두는 개념이라고 생각하면 됩니다.
:::

## 세 가지 축으로 이해하기

OpenClaw의 실행 안전성은 아래 세 축으로 나눠 보면 이해가 쉽습니다.

### 1. sandbox mode

도구 실행을 로컬 호스트에서 할지, 더 격리된 환경에서 할지 정하는 층입니다. 개발 편의성과 격리 수준의 균형을 잡는 설정이라고 보면 됩니다.

### 2. tool policy

어떤 도구를 허용하거나 막을지, 특정 도구를 어떤 조건에서만 쓰게 할지 정하는 층입니다. 예를 들어 파일 읽기는 허용하지만 셸 실행은 제한할 수 있습니다.

### 3. elevated 실행 구분

같은 `exec`라도 일반 권한과 **elevated** 실행은 위험도가 완전히 다릅니다. 높은 권한 실행은 별도 승인 경계로 다루는 것이 맞습니다.

## 왜 이 구분이 중요한가

실제 위험은 대부분 “도구가 있느냐”보다 “도구가 어떤 권한으로 어느 환경에서 실행되느냐”에서 발생합니다.

예:

- `read`만 허용 → 비교적 낮은 위험
- `exec` 허용 → 환경에 따라 중간~높은 위험
- `exec` + elevated 허용 → 매우 높은 위험
- `browser`로 로그인 세션 접근 가능 → 데이터 민감도 상승

그래서 샌드박싱은 단독 기능이 아니라, **도구 카탈로그와 권한 체계 위에 올라간 정책 레이어**입니다.

## 운영 패턴

### 개발/개인 환경

- 빠른 반복이 중요
- 로컬 실행 비중이 높음
- 다만 민감한 명령은 정책으로 좁히는 것이 좋음

### 공유 서버/장기 운영 환경

- 격리 우선
- 파일 시스템 범위 제한
- 네트워크 접근 최소화
- elevated 실행 최소화

즉, 서버형 Gateway일수록 샌드박싱은 옵션이 아니라 **운영 기본값**으로 보는 편이 안전합니다.

## tool policy가 다루는 것

tool policy는 보통 아래를 다룹니다.

- 사용 가능한 도구 목록
- 특정 도구의 파라미터 제한
- 쓰기/삭제/실행 계열 도구의 승인 경계
- 워크스페이스 또는 대상 경로 제한
- 고위험 옵션(`elevated`, 외부 네트워크, 사용자 프로필 접근 등) 제한

이 정책은 [도구 샌드박스](/tools/sandbox) 문서에서 더 구체적으로 설명합니다.

## elevated는 별도 위험으로 보세요

`elevated`는 단순한 옵션이 아니라, “호스트 권한 경계를 더 높게 넘는다”는 뜻입니다. 따라서 다음 원칙이 좋습니다.

- 꼭 필요한 경우에만 사용
- 승인 가능한 정확한 명령만 실행
- 광범위한 체인 명령은 특히 신중하게 검토
- 서버 운영 환경에서는 가능한 기본 비활성화

## 운영 점검

아래 명령어를 터미널에 입력하세요:
```bash
openclaw gateway status
openclaw doctor
```

- `openclaw gateway status`: 실행 환경과 기본 상태를 확인합니다.
- `openclaw doctor`: 샌드박스/정책/환경 이상을 함께 점검하는 출발점입니다.
