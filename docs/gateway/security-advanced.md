# 보안 상세 가이드 (Advanced Security)

이 문서에서는 OpenClaw Gateway를 **더 안전하게 운영하는 실전 방법**을 배웁니다.

## 📌 이 문서에서 배우는 것
- Sandbox 모드를 어떻게 이해하면 좋은지
- Tool Policy로 도구 권한을 어떻게 좁히는지
- SecretRef, 환경 변수, `secrets.json` 차이
- `openclaw security audit`를 어떻게 읽는지
- 실전 보안 체크리스트

보안은 어려워 보여도, 사실은 **문을 여러 겹으로 잠그는 일**이에요.
한 개만 믿지 말고, 여러 개를 함께 쓰는 것이 핵심입니다.

:::tip 💡 쉽게 이해하기
보안은 **현관문 + 비밀번호 + 경비실 + 금고**를 함께 두는 것과 비슷해요.
한 가지가 실수해도, 다른 장치가 막아 주면 더 안전합니다.
:::

## Sandbox 모드 자세히 보기

Sandbox는 에이전트가 도구를 쓸 때 **어디에서**, **얼마나 안전하게**, **얼마나 큰 힘으로** 실행할지 정하는 장치입니다.

### Sandbox를 왜 쓰나요?

도구가 강할수록 사고가 나면 피해도 커집니다.
예를 들어:

- `read`만 가능하면 비교적 안전함
- `write`가 열리면 파일이 바뀔 수 있음
- `exec`가 열리면 명령어를 실행할 수 있음
- `elevated`까지 열리면 더 강한 권한을 쓸 수 있음

그래서 Sandbox는 **도구를 막는 장치**이기도 하고,
동시에 **사고가 나도 범위를 작게 만드는 장치**이기도 합니다.

### 이해를 위한 3단계

#### 1) 실행 위치
- 로컬 호스트에서 바로 실행할지
- 더 격리된 공간에서 실행할지

#### 2) 허용 범위
- 어떤 도구까지 허용할지
- 어떤 경로까지만 읽고 쓸지

#### 3) 높은 권한 여부
- 일반 권한만 허용할지
- `elevated` 같은 높은 권한은 막을지

### 안전한 기본 생각

- 개인 로컬 테스트 → 편의성도 중요
- 서버 운영 → 격리와 제한이 더 중요
- 공개 환경 → 최소 권한 원칙이 가장 중요

:::tip 추천 기본값
처음에는 **읽기 중심**, **쓰기 최소화**, **`exec` 제한**, **`elevated`는 기본 차단**으로 시작하면 안전합니다.
:::

## Tool Policy: 도구별 허용/차단 정책

Tool Policy는 말 그대로 **어떤 도구를 쓸 수 있는지 정하는 규칙표**입니다.

쉽게 말하면:
- 연필은 허용
- 가위는 감독할 때만 허용
- 열쇠는 관리자만 사용

같은 느낌이에요.

### 왜 중요한가요?

OpenClaw는 아주 다양한 도구를 연결할 수 있습니다.
그중 어떤 도구는 비교적 안전하고, 어떤 도구는 매우 강력합니다.

예를 들어:

| 도구 | 위험도 느낌 | 이유 |
|---|---|---|
| `read` | 낮음 | 읽기만 하기 때문 |
| `write` / `edit` | 중간 | 파일 내용을 바꿀 수 있기 때문 |
| `exec` | 높음 | 셸 명령을 실행할 수 있기 때문 |
| `browser` | 중간~높음 | 로그인 세션, 웹 데이터에 닿을 수 있기 때문 |
| `message` | 중간~높음 | 외부 채널에 내용을 보낼 수 있기 때문 |

### Tool Policy에서 자주 제한하는 것

- 특정 도구 자체 차단
- 특정 경로 밖 파일 접근 차단
- 삭제/쓰기/실행 계열 제한
- 외부 네트워크 사용 제한
- 사용자 브라우저 프로필 접근 제한
- `elevated` 옵션 차단

### 실전 운영 패턴

#### 개인 실험용
- `read`, `write`, `edit` 중심
- `exec`는 필요할 때만
- 위험한 외부 전송 도구는 최소화

#### 팀/서버 운영용
- 기본은 읽기 위주
- 쓰기와 실행은 승인 기반
- 외부 발송 도구는 더 엄격하게

:::tip 기억하기 쉬운 원칙
**도구가 강할수록 더 좁게 허용한다**.
이 한 줄만 기억해도 큰 실수를 많이 줄일 수 있어요.
:::

## Secrets Management: 비밀값 관리

비밀값(secret)은 API 키, 토큰, 비밀번호처럼 **남에게 보이면 안 되는 정보**입니다.

OpenClaw에서는 보통 아래 3가지를 자주 보게 됩니다.

### 1. SecretRef

SecretRef는 설정 파일 안에 비밀값을 직접 적지 않고,
**다른 안전한 저장 위치를 가리키는 표식**입니다.

장점:
- 설정 파일에 비밀이 덜 드러남
- 비밀 교체가 쉬움
- 공유 저장소에 실수로 올릴 위험이 줄어듦

쉽게 말하면 **보물 상자 주소만 적어 두는 방식**이에요.

### 2. 환경 변수 (Environment Variables)

환경 변수는 프로그램이 실행될 때 바깥에서 값을 넣어 주는 방식입니다.

장점:
- 코드나 설정 파일에 직접 키를 덜 남김
- 서버, systemd, 컨테이너에서 관리하기 쉬움

주의:
- 쉘 기록, 로그, 프로세스 노출에 주의
- 여러 사람이 보는 화면에 그대로 출력하지 않기

### 3. `secrets.json`

`secrets.json`은 비밀값을 모아 두는 파일 방식입니다.

장점:
- 한 곳에 모아 관리하기 쉬움
- 로컬 개발에서 간단함

주의:
- 파일 권한을 꼭 제한해야 함
- Git에 올리지 않도록 주의해야 함
- 백업 파일이 다른 곳에 남지 않게 확인해야 함

### 무엇을 먼저 쓰면 좋을까요?

보통은 이렇게 생각하면 쉽습니다.

1. 가능하면 **SecretRef** 우선
2. 운영 환경에서는 **환경 변수** 함께 활용
3. 단순 로컬 환경에서는 `secrets.json` 가능

### 하면 안 좋은 것

- API 키를 `openclaw.json`에 그대로 적기
- 비밀값을 문서 예시에 실제 값으로 넣기
- 스크린샷에 토큰이 보이게 두기
- Git 커밋에 비밀값 포함하기

:::tip 좋은 습관
비밀값은 **설정 파일에 직접 쓰지 않는다**고 생각하면 대부분의 사고를 줄일 수 있어요.
:::

## `openclaw security audit` 자세히 보기

아래 명령어를 터미널에 입력하세요:

```bash
openclaw security audit
```

이 명령은 OpenClaw 설치 상태를 보면서 **어디가 느슨한지** 알려줍니다.

### 어떤 것을 볼 수 있나요?

보통 이런 항목을 점검합니다.

- 인증이 꺼져 있는지
- 비밀값이 설정 파일에 직접 들어 있는지
- 도구 권한이 너무 넓은지
- 샌드박스가 약한지
- 원격 노출 방식이 위험한지

### 결과를 읽는 순서

#### 1) 높은 위험부터 보기
먼저 **Critical**, **High**, **Warning**처럼 강한 경고부터 봅니다.

#### 2) 바로 고칠 수 있는 것 먼저
예:
- 인증 켜기
- 공개 bind 줄이기
- 비밀값 이동
- `exec` 범위 축소

#### 3) 다시 audit 실행
고친 뒤 다시 실행해서 경고가 줄었는지 확인합니다.

### audit와 doctor 차이

- `openclaw security audit` → **보안 중심 점검**
- `openclaw doctor` → **전체 운영 점검**

둘은 경쟁 관계가 아니라, 같이 쓰면 더 좋습니다.

:::tip 추천 흐름
1. `openclaw security audit`
2. `openclaw doctor`
3. `openclaw gateway status`

이 3개를 함께 보면 **보안 + 운영 상태**를 같이 볼 수 있어요.
:::

## 실전 보안 체크리스트

아래 체크리스트를 위에서 아래로 보면 됩니다.

### 1. 네트워크
- Gateway는 가능하면 loopback 주소에 바인딩
- 인터넷에 바로 공개하지 않기
- 필요하면 Tailscale이나 신뢰된 프록시 사용
- `allowFrom`으로 접근 가능한 출발지 줄이기

### 2. 인증
- `auth.mode` 활성화
- 공개 URL + 무인증 조합 피하기
- 토큰은 짧게, 명확하게, 안전하게 보관

### 3. 도구 권한
- 꼭 필요한 도구만 허용
- `exec`, `browser`, `message`는 범위를 좁게
- `elevated`는 기본 차단하고, 정말 필요할 때만 허용

### 4. 비밀값
- SecretRef 또는 환경 변수 사용
- `secrets.json` 파일 권한 제한
- Git, 로그, 스크린샷에 노출 금지

### 5. 채널/노드
- DM pairing 우선
- 그룹 반응 범위 줄이기
- 모바일/노드 연결 시 승인 흐름 확인

### 6. 점검 루틴
- 변경 후 `openclaw security audit`
- 운영 중 `openclaw doctor`
- 정기적으로 설정 재검토

## ✅ 성공 확인

아래 상태에 가까우면 잘 되고 있는 것입니다.

- `openclaw security audit` 경고가 줄어듭니다.
- 비밀값이 설정 파일 본문에서 사라집니다.
- 강한 도구가 꼭 필요한 범위로만 줄어듭니다.
- 원격 접속이 인증과 제한 속에서 동작합니다.
- 운영자가 “무엇이 열려 있는지” 설명할 수 있습니다.
